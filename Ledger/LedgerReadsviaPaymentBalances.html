Ledger Reads via PaymentBalances

Author: Daniel Skarbek

Date: Mar 22, 2023

Status: Draft

Goal

We want Books service to be completely encapsulated by FinancialAccount[a][b] (which is the platform service under PaymentBalances).  That means all connectivity between Ledger and Books needs to be removed.  The plan is to first replace all reads with calls to PaymentBalances and then replace all writes.  This doc is about removing read connections from Ledger to Books, and we also are including feed consumption in that scope.

How to Replace the Books feed with the PayBal feed

The BooksJournalCommandFeedListener currently listens to the feed of journal commands from books to find activity to replay on ledger.  It filters for stuff that has an accounting event (Bizbank sweep stuff won’t) and isn’t flagged for “skip ledger” (only the Bizbank events and Deposit Fee events will be).  As a reminder, the current system is a “either one first” system where each system reads the feed of the other to ensure that everything ends up on both systems.  Here’s an old diagram of the design that shows a ledger-first BCE in blue (only Bizbank and Deposit Fee events now) and a books-first AE in green.  Follow the green and blue flows to see how the two different types of events propagate through the system.

We started with a bunch of handlers that went to ledger first and converted most of them to be books-first.  Now we want those books-first events to flow into the PayBal feed to get applied to ledger instead of being picked up from the journal command feed.  So, we can’t just have the kafka feed start processing all activity all of a sudden, we’re going to roll out kafka feed handlers event by event, and the few events that are still ledger-first shouldn’t be picked up at all yet.  So, the PB activity feed has to filter based on event type.

We also need some change-over time because things are asynchronous here.  We need to turn on PB activity feed processing for an event and then after a pause turn off the books feed processing of that event.  In the interim, the event will be double processed, but idempotency in ledger will protect us there.  It will also be important that the integration tests we have in ledger are verifying that each event type does get processed in both books and ledger.

Alternatively, we could have an explicit flag in the PB activity that says whether we are expecting the activity feed to apply it to ledger or not, but that seems unnecessary given that we can idempotently process from two sources during the change-over.

My recommendation is that the ledger consumer of the PB activity kafka feed should have a map by activity type of handlers that it will dispatch to.  Then each of these handlers should have a “will process” method that can determine if this specific activity of that type should be processed or not.  Eventually this can just return true, or be factored out, but for a while it will be useful as a place to have a feature flag for enabling that handler, or as a place to discern between handled and unhandled activities of the same type.  Then, if the “will process” check passes, then a process method is called which should call the same ledger processing logic that the books feed handler uses.  Once we have a new handler like this ready to go for an event, we can then change the AE handler, that writes to books in the first place, to set the “skip ledger” flag on the AE that it embeds in the journal command.  That will stop the books feed handler from processing it.  This is probably switched with a feature flag, but not the same feature flag as the one turning on the new handler.  The new handler needs to be turned on – wait a little – then turn off the old books feed handler.

How to Convert the Handlers

See: Payment Balance Activity type mapping (go/pbactivitytypes) for mappings of types

The current system has books and ledger handlers that both work off of an input AccountingEvent to create potentially multiple ledger or book entries for that event.  We want to switch to a system where the book handlers (AccountingEventHandlers) create journal entries from the AccountingEvent and then we have activity feed handlers that convert each activity (basically the JournalEntry) into ledger entries.  Most of the time this will be a one to one conversion, but some cases like Disputes will have some gotchas.  Most importantly this means that rather than re-calculating results from the AccountingEvent, all the results should be present in the JournalEntry / Activity and we should be using the same result directly.

We specifically want to enforce that the AccountingEventHandler creates the ledgerEntryToken and passes it through and the ActivityHandler just uses what it is given.  For other data that is being passed through to be used, we may have to add additional data to the type detail structures in the PB Activity, but this means that they also get added to the PB API and to the PayoutEntries API which is GA’ed.  So, we need to be thoughtful about what and how exactly data gets exposed.  If needed, we may need to expose some things only in the “internal properties” part of the activity proto so that they aren’t seen externally.  Each piece of data needs to be carefully considered.
