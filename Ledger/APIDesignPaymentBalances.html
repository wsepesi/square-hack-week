Overview

This is the API for the core balance and account concept behind payment processing and payouts to sellers.  This doc is looking only at the read endpoints that would be used by external clients.  The rest of the endpoints that would only be exposed internally are in the sibling doc for the write side of the API.  But, not looking to get formal approval of that yet.  Initial release will be just this part of the API.[a]

Resources

The Payment Balances API will include the following resources. Read-only or server-generated fields are bold.

PaymentBalanceAccount

Represents an account of money from payment processing. A seller (location) may have more than one for different currencies or uses.  An Account contains Balances. The other uses, or types, of account are for things like having fees recorded in a separate account that is settled independently, or for having a risk reserve account.  The PAYOUT account is the general account for money that is available to be paid out.  (As opposed to being taken for fees or held as a risk reserve.)

PaymentBalanceAccountBalance

A balance of an account.  As transactions happen against an account, the account is not mutating, its balance is.  The primary balance of an account is its POSTED balance, this is the balance of what has actually been committed or posted to the account.  But, an account can also have available and pending balances that expose the effect of pending transactions that are in-progress and may or may not end up “landing” on the official / posted balance.  All balances are implicitly created when the account is created.  The balances of the account are singleton resources which are all modeled by this object.

ListPaymentAccounts

GET v2/payment-balances/accounts

Permissions: PAYMENT_BALANCES_READ

Find the payment account(s) for a seller / location, potentially based on type and currency.  This is a discovery operation for clients that don’t know the account id they want to work with.  We support pagination to ensure that there isn’t an unbounded query here, but in reality, each location will have only a few accounts.  So, the limit should be rarely, if ever, hit.

Name

Type

Description

errors

repeated Error

Information about errors encountered during the request.

starting_balance_money (may not be in alpha)

Money

The balance amount as of just prior to the start_time.  On subsequent pages, the same amount is returned here; it’s the starting balance of the whole, paginated list.

ending_balance_money (may not be in alpha)

Money

The balance amount as of the end_time.  On subsequent pages, the same amount is returned here; it’s the ending balance of the whole, paginated list.

activities

repeated PaymentBalanceAccountBalanceActivity

The ordered list of balance changes that make up the history between the start_time and end_time of the balance.  Ordering is by available_at in ascending (oldest to newest) order.  The net_money and running_balance_money amounts of the BalanceChange in aggregate will show the transformation from starting balance to ending balance.  If there are more balance changes than the limit allows, then a cursor for the next page will be returned.  But, even if pagination occurs, the result is guaranteed to be consistent.  Meaning, new activity, even if back-dated, won’t change the response.  New activity will be ignored.  You are essentially reading a snapshot.

string

When a response is truncated, it includes a cursor that you can use in a subsequent request to fetch the next set of results. If empty, this is the final response. For more information, see Pagination.

Security

Authentication & Authorization

Authentication and authorization of external clients is done via OAuth, which enforces that an oauth client (ie. application) has authorization to operate on a seller account’s data while also authenticating the application’s identity. New OAuth permissions:

  • PAYMENT_BALANCES_READ
  • PAYMENT_BALANCES_WRITE

First-party platform applications will be authorized via Multi-Target Tokens to validate the client permissions.

PII + Data Privacy

The Payment Account service will not directly store any PII or confidential data, only tokens to data in other services.

It is unclear whether the transaction data in the underlying Financial Account is in scope for relevant privacy regulations. If so we may need to provide for account deletion capabilities to provide account service in applicable jurisdictions.

None of the data directly stored by the Payment Account service should have privacy implications.

Competitive Analysis

We have taken a lot of inspiration from Stripe’s relevant APIs.  We also looked at the CorePro APIs, but didn’t lean on them much as we felt they were less relevant.  Stripe on the other hand is also directly solving the exact same challenges we are looking at and our APIs look substantially similar[x][y].  One current difference we considered was in naming.  What they call “balance transactions” we called “balance activity.”  This is to be consistent with the CustomerBalance API design, and “Transaction” is very overloaded, both in the financial realm and in the computer science realm.  “Balance activity” means the same thing in this context and is simpler and less likely to be conflated with other concepts.  Also, our further (to be defined) write API will define an Account Transaction that belongs to the account and are distinct from the Balance Transaction concept.

Alternative Designs

One of the big trade-offs in the design was whether to expose what I would call an “account model” or a “balance model”.  By “balance model” I mean that the basic entity is a balance, it has a single amount and changes and users are interacting with balances.  By “account model” I mean that the basic entity is an account that has multiple balances, each with an amount and history of changes.  You can have whatever accounts you want, but each account has a specific, enumerated set of balances.  Primarily these are “posted” and “available.”  The Financial Account primitive we are built on top of has an account model where each account has a posted, current, pending_in, and pending_out balance.  These balances are all related and are changed by state transitions in transactions against the account.  Meaning a pending transaction may affect the pending and available balance of the account, but not affect the posted balance until the transaction is finalized.  Or, if it is canceled then there is a reverse change put on the available and pending balance and no change ever to the posted balance.  So, an account model tends to have these alternate views of the balance and a concept for pending transactions, where-as a balance model is simpler and just has one balance (the equivalent of the posted balance in the account model).

The trick is that we have two very different sets of users.  We have third party users that are primarily using the read APIs to do reconciliation.  This usage really only needs a balance model and we wanted to keep the API as simple as possible for these users.  At the same time we have first party users that are doing reads and writes and the underlying mechanic of the system is an account model with transactions that are pended and then posted, so we need to expose some of that for our first party users.  So, we are torn between two different sets of users and wanting to have one, external ready api that isn’t overly complicated or under-powered.

The compromise we have proposed here is some-what of a mixed model.  We expose the account concept mostly as the immutable part that holds a balance, but it also gives us a hook to get into that model if we need to.  On the balance part, we expose the balance type issue but will document that most everyone probably just wants the posted balance.  The account transaction concept will be hidden from external users.  As we develop the interaction for write use cases (which is entirely an internal capability) we will give them access to the account transaction concept and it should dove-tail pretty well with what we’ve exposed externally.  We considered having parallel end-points for the external users that don’t expose balance type even and just show the posted balance (making the external model much more balance oriented).  But, this would mean having redundant end-points in the model, one for external and one for internal, and there was one external customer that expressed an interest in seeing the pending balance.

[t]I know this is an old comment, but happened to read it again and wanted to call out the general comment that "name1/name2/:name2id" is not that uncommon when you assume "name1" is a domain name, as it is here.  This is fairly standard REST practice to the best of my knowledge.  It's hard to have all your resources at the root level and avoid naming conflicts so you tend to group resource into domains.  In the payment-balances domain we have resources "accounts" and "transactions".  We then also have other resources that are organized under these resources, like "balance" and "activities".  So, maybe they are getting thrown off that there are two names in a row in the path without an id for the first name, but "payment-balances/accounts" isn't an instance of one resource being nested under another, it's a resource scoped by a domain.  IMO this is totally normal, good REST-ful design.
