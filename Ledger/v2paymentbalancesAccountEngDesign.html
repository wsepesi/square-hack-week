Overview

Payment Balances service enables the v2/payment-balance api for both first and third parties to understand sellers’ core balance and account concept, with Financial Account primitives as the underlying accounting model interacting with Books, a double entry accounting system.

This doc focuses on the eng design of Payment Balances account read operation for existing Ledger Books.

Audience

  • Accounting Engineering - Financial Account team

Terminology

Financial Account primitives

Background & Project Overview

Currently sellers balance and their payment activities are persisted both in Ledger SQL db and in Books, a double entry accounting system. We are migrating the entire accounting system to treat Books as the source of truth to decompose ledger service.

Ledger now directly interacts with Books service. To facilitate interaction with Books and formailize payment and balance models, the end result should be Ledger calling v2/payment-balances API, which calls Financial Account service to interact with Books.

v2/payment-balances API provides READ on account, balance, and balance activities information. In the near future, the v2/payment-balances API will also provide privileged access for internal clients to write to Books through the Financial Account.

The entire project includes 1) enabling READ from v2/payment-balances on existing Ledger Books for account, balance and balance activities. 2) Reorganize Ledger Books schema to the designed Financial Account schema. 3) Enable privileged parties to write to Books via v2/payment-balances API.

This doc will only focus on part 1) and specifically only on reading Payment Balance accounts via v2/payment-balances APIs. As all existing Ledger books do not have mapping Payment Balance Accounts (PBAs) and Financial Accounts(FAs) yet, we will first backfill all Ledger merchants with PBA and FA, and we will create PBA and FA at new Book creation time as well.

Goals

  • All existing and new Ledger merchants have corresponding Payment Balance accounts(PBA) and Financial Accounts(FA).
  • Clients could read PB Accounts by calling v2/payment-balances APIs
  • ListPaymentAccounts
  • RetrievePaymentAccount.

Non-Goals

  • Enable READ from v2/payment-balances on Balance and Balance Activities. This design will be elaborated in the next design doc.
  • RetrievePaymentAccountBalance
  • ListPaymentAccountBalanceActivities
  • RetrievePaymentAccountBalanceActivity
  • Reorganize Ledger Books schema to Financial Account schema
  • Enable privileged WRITE account operation from v2/payment-balances API

Eng Design

Introduction

In Ledger, each unit_token, which uniquely identifies a location of a merchant, has a set of books. Among all these books, only a couple are externally exposed, while the rest are playing the role of “contra” book which net zero the balance in the double-entry accounting system. Going forward with the payment balances model, each externally exposed “posted” book maps to a Payment Balance Account(PBA) of different types(e.g. PAYOUT, FEE, RISK_RESERVE). After corresponding PBAs and FAs are created from existing books, Ledger will store mapping between a unit and its PBA ids.

Payment Balance account and Financial Account is 1 to 1 mapping. Following the Financial Account primitive model, each FA has five books: Posted, Current, Pending in, Pending out, and Contra. But for existing Ledger Books, not all five books existed. For `PAYOUT` type PBA, `merchant_payable` book maps to posted_book, and current_book of FA, while `payment_pending_risk` book maps to pending_in_book of FA. For `FEE` and `RISK_RESERVE` type PBA, `fee_payable` and `risk_reserve_payable` books are both current_book and posted_book of respective FA.

Currently Ledger directly reads and writes to Ledger Books. To migrate Ledger to call v2/payment-balances, which calls the Financial Account to interact with Books, we need to backfill all existing units with PBAs and FAs, and create new accounts when a new Book is created. We will store unit token and its PBA ids in Ledger. The key design principle here is keeping the mapping out of PB in order to maintain flexibility and avoid coupling PBs to their related entities.

Account creation and backfill in Ledger

Below are the steps for creating a new PBA + a new FA when Ledger either creates a new Book or backfills a merchant:

  1. If Ledger does not have PBA ids for a unit, either the merchant is just newly created, or a backfill is needed. Ledger will call Books to get the relevant Book ids for PB creation. For now, we will focus on PB with types of PAYOUT, FEE, and RISK_RESERVE, which use merchant_payable, fee_payable, and risk_reserve_payable Book as the posted_book of corresponding FA.
  2. Ledger calls PB with the Book id(s) and other information needed to create PB and FA.
  3. PB calls FA to create a new Financial Account with existing book ids.
  4. Newly created FA is returned to PB, and PB creates PaymentBalanceAccount from FA info.
  5. PB returns the PBA id back to Ledger, where ledger stores the relationship between the unit_token and PB account id, as all other APIs to interact with v2/payment-balances require the PB account id.

  1. Asynchronously, PB consumes a feed FA publishes on account creation and update.
  2. PB transforms the FA info into PBA and async publishes the PBA to EG.

  1. When PB calls FA to create an FA, an idempotency key is needed to avoid duplicated FA creation.
  1. We will use <book owner token> + pb_type + currency combination to uniquely identify a Financial Account. This combination should only map to one PBA/FA. Note that a book owner token can be either a unit_token or rarely a merchant_token.

PB will utilize the same encoding function mentioned above(adding a prefix) to get the PBA id out of the FA id. Also, PB will denormalize fields in FA metadata to create the PBA object, and store the PBA in EG.

This mapping table will be on master shard so that it could serve mapping both directions between a `payment_balance_account_token` and combination of `merchant_id` + `payment_balance_account_type`.

  1. What are the fields in this table?

In the medium term, Ledger will at most have three types of payment balance accounts(Payout, Fee, and Risk_Reserve as listed).

The `merchant_id` is the foreign key to the `merchants` table. The backfill job will go through the `merchants` table and store the existing mapping in this `payment_balances` table.

  1. Why store this mapping in Ledger?

It's important that the mapping not be stored directly on the PaymentBalance object itself.  Doing this would couple the payment-balance with the definition/scope of the balance’s owner(similarly to how a Unit is presently coupled/conflated with a Balance).  

Ultimately, we see a separate domain object responsible for configuring how payments get processed against a balance and that that object would exist outside of ledger.  In the meantime, ledger is a natural place to cache the mapping as it requires the mapping to process payments.

  1. Backfill job steps [ACTENG-4412]:
  1. Get a token from `merchant` table or from the backfill endpoint
  2. Call Books to find all externally exposed books(merchant_payable, payment_pending_risk, fee_payable, and risk_reserve_payable books) belonging to this merchant/unit token.
  3. If no books existed, no-op the rest of backfill action. If the listed books partially existed, store the existing PBA ids in the `payment_balances` table, and leave the rest PBA ids as null.
  4. For merchant_payable and payment_pending_risk books, create a PAYOUT type PBA, and FA with posted_book_id = merchant_payable id, current_book_id = merchant_payable id and pending_in_book_id = payment_pending_risk book id.

For any existing fee_payable book, create a FEE type PBA and FA with posted_book_id = fee_payable book id, and current_book_id = fee_payable book id.

For any existing risk_reserve_payable book, create a RISK_RESERVE type PBA and FA with posted_book_id = risk_reserve_payable id and current_book_id = risk_reserve_payable book id.

  1. Call MDS to find out if the token is a merchant or unit token, and if the token belongs to a unit, find its merchant_token.
  2. Call PB endpoint with correct PB type + book ids, and additional metadata info like unit_token, merchant_token, currency etc.
  3. Ledger gets back PB ids from PB endpoint, and stores the merchant token to PBA id mappings in Ledger table. 

  1. Ledger calls PB to create a PBA and FA while creating new Books, then stores the PBA ids in the Ledger table. [ACTENG-4411]
  1. When Ledger finds a new unit, Ledger creates default books(merchant_payable, and payment_pending_risk). After the Books are created, we need to call PB endpoint to create PAYOUT type PBA, with posted_book_id = merchant_payable book id, and pending_in_book_id = payment_pending_risk book id.
  2. Currently, Fee_payable books get created in Ledger Journaler when a cost plus fee is to be persisted in Books. We need to first add code in Ledger before calling AQ, to create a fee_payable book if a costplus merchant does not have an existing one.

Once the books are created and book id is returned, call PB endpoint to create FEE type PBA, with posted_book_id = fee_payable book id, and current_book_id = fee_payable book id as well.

  1. Currently, Risk_reserve_payable books also get created in Ledger Journaler when a risk reserve is to be held or released. We need to first add code in Ledger before calling AQ, to create a risk_reserve_payable book if a risk reserve merchant does not have an existing one.

Once the books are created and book id is returned, call PB endpoint to create RISK_RESERVE type PBA, with posted_book_id = risk_reserve_payable book id, and current_book_id = risk_reserve_payable book id as well.

API

This design focuses on READ requests related to Payment Balance Accounts.

It includes two APIs:

ListPaymentAccounts

RetrievePaymentAccount 

Architecture of API calls on accounts read

After backfill work mentioned in the previous section is done, clients of PB will have all the Payment Balance Account(PBA) ids and their unit_token stored on client side. There are two ways clients could get the full PB information by calling v2/payment-balances.

RetrievePaymentAccount 

The client will call v2/payment-balances on RetrievePaymentAccount with a PBA id. FA is the source of truth for PBA info, thus PB will call the Financial Account service and transfer response from FA into PBA and return the PBA to the client.

ListPaymentAccounts 

Clients could call v2/payment-balances on ListPaymentAccounts API with either a merchant_id, or a unit_token + PB type + currency combination. Because PB is stateless and FA does not have  indexes on these properties but only on FA id, PB needs to call ElasticGraph for the corresponding PBA(s).

Note PB information in EG has eventual consistency, so this API call should expect some latency in account creation and update. This is acceptable as we foresee this API will not have high traffic nor of high priority in application.

Alternative Designs

We’ve decided to make Payment Balance stateless. This is based on the usage of Elastic Graph, adding metadata in Financial Account, having PB account id an encoded version of FA id, and client storing the entity<-> PB account id relationship. We foresee there will be a centralized system to store the mapping between a PB owner and their PBA ids.

The alternative design is to have a DynamoDB in PB service. It centralizes the information mentioned above but adds another hop in each call and adds dependency to DynamoDb. The alternative design of the datastore is as below.

Datastore

For Payment Balances, data will be stored in DynamoDB. Payment Balances(PB) service will only store mapping between legal external entities and payment balances accounts of different types and currencies.

This mapping may be able to move further upstream in MDS in the future so clients will call PB service with the account id and the service could get rid of any data storage. But for the current stage, we want to enable Ledger to call Payment Balances service with unit_token and create the account on the fly, and storing such mapping in PB makes the most sense.
