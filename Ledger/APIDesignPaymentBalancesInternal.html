Overview

This is the extension of the API Design - Payment Balances to add the privileged end-points, meaning end-points that we wouldn’t expose to external users.  This includes all writes, and also reads related to entities that are only interacted with when doing writes.  This doc is an extension to the base level API Design.

Resources

These are the additional resource types that are used in these end-points (beyond the resources already defined in the API Design - Payment Balances).

PaymentBalanceTransaction

Represents a single money movement into or out of an account.  Potentially has a short life cycle where it is first pending and then posted or canceled.

The amount of fees that were charged (if MONEY_IN) or refunded (if MONEY_OUT) on this activity, prior to the activity being recorded on the balance.  (This does not include fees or adjustments that are recorded as separate balance activities.)

The gross amount for the activity before fees were taken out.  Gross_amount_money minus fee_amount_money equals net_amount_money.

The state of this transaction.

PENDING means that we’ve reserved the amount, but haven’t committed the change yet.

POSTED means we’ve committed the change to the posted balance of the account.
CANCELED means we’ve released the reserved funds and have not made any change to the posted balance of the account.
FAILED means that the attempt to reserve funds failed.

PaymentBalanceActivity<type>Detail

Polymorphic sub-object will be defined for each type value that has additional fields.  Same as defined on the PaymentBalanceActivity and copied to that object on creation.

idempotency_token

string

Two duplicate requests with the same idempotency_token will always return the same result and have the same effect as only one of them being submitted.

state

enum

{PENDING, POSTED}

POSTED will do a 1-phase transaction that is immediately complete.  PENDING will start a 2-phase transaction that must be followed up with a POSTED or CANCELED status.  Note that clients are responsible for ensuring that pending transactions are resolved.  We will leave them pending forever otherwise.  A PENDING can fail on MONEY_OUT (resulting in a FAILED status) if there is not enough money available in the account.  A POSTED status will allow the account to go negative.[f]  In the case of a FAILED transaction, the transaction is still created, but a 409 code is returned.

net_amount_money

Money

Must be positive.  The amount to either PEND or POST.

fee_amount_money (Optional)

Money

Must be positive.  The amount of fees that were charged (if MONEY_IN) or refunded (if MONEY_OUT) on this activity.  Assumed zero if omitted.  This does not change the effect on the balance, but does help indicate what the gross amount of the transaction was before fees were removed by giving the amount of fees removed.  It is assumed that the gross amount is net_money plus fee_money.

originated_at (optional)

string

The timestamp, in RFC3339 format, for when the event that caused this transaction happened.  Defaults to “now”.

RetrievePaymentAccountPendingInBalance

GET v2/payment-balances/accounts/{account_id}/pending-in-balance

Permissions: PAYMENTBALANCES_WRITE

Get the “pending in” balance for the account which is the amount that is currently waiting in pending MONEY_IN direction transactions.  This operation takes the exact same path/query parameters and returns the same object as the RetrievePaymentAccountBalance operation except that it is the pending-in balance data instead of the posted balance data.

[a]I haven't came up with an example yet but would there be a Financial Account transaction(assuming that's what the transaction here means) that changes multiple payment account balances?

[b]No.  They are 1:1 and the FinAct transactions only touch a single FinAct.  I think we may eventually want to define a "transfer" operation that moves money between payment balances, but even that would create 2 transactions for you.

[c]Great thanks!

[d]how does this value change if gross settlement is enabled/disabled?

[e]This is the amount by which the balance has changed.  In a gross settlement case the fee_money will be zero and the gross_money will be the same value.  In a net settlement scenario, for the payment capture, there may be a fee_money and a different gross_money.  The net_money is THE amount, but if that amount was the result of some calculation that adjusted some other amount, you can expose that by giving the prior amount as the gross_money and then the fee_money should be the difference between them.  This would only be used for fees that are not reported as their own balance activities.

[f]The FinancialAccount API allows much more nuanced control here, but I think we can simplify down to this rule that PENDs are for when you want to guarantee the funds.  Otherwise you would just use a POST.  I'm assuming that since this api is just used internally we can change it as we go.  As we convert ledger to use it, we will discover all the use-cases and if we find something that needs a more complicated API we could retrofit it (in a non-backwards compatible way) since it isn't publicly exposed.  If we need to ensure we don't do that, we would need to do a whole lot more research on use-cases before we committed to the API.
